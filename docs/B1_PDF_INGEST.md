# B1 PDF Ingest

Documentation: [A0 Documentation Index](./A0_DOC_INDEX.md)

This document explains how downloaded PDFs are processed, which artifacts are generated by default, and which artifacts are generated only on demand.

## 1. Default behavior

Downloading a PDF alone does not trigger parsing.

PDF ingest runs only when the user requests reading, analysis, search, summarization, evidence extraction, or follow-up question answering.

## 2. Core ingest pipeline

```text
PDF
-> compute MD5 / SHA-256
-> compare the SHA-256 value with the value in the existing meta.json
-> reuse pages.jsonl when the hashes match; reparse when they do not
-> generate meta.json
-> extract text page by page with PyMuPDF
-> generate pages.jsonl
-> generate quality_report.json
-> write to the SQLite document_pages and document_pages_fts tables
```

All of these steps run locally in Python and consume no LLM tokens.

Annual reports and prospectuses downloaded from official sources first enter `staging/downloads/`. A single page-level extraction pass validates structural completeness, company or stock-code identity, and reporting year. The file moves to `raw/` only after it passes validation. If analysis is also requested, the extracted pages are passed directly to ingest so the entire PDF does not need to be parsed again after the completeness check.

Successful validation generates `validation_report.json` and binds it to the document ID, SHA-256 hash, stock code, and document type. When an official source is explicitly refreshed, the record is reused if the local PDF associated with the URL has not changed. Extraction and validation must run again if the file hash changes, the identity does not match, or the previous validation did not pass.

Incomplete candidates are deleted from staging before ingest. Scanned files that cannot be assessed reliably, or candidates with identity anomalies, are retained in `staging/review/` for OCR or manual review. Existing files in the production `raw/` directory are never deleted automatically because of a single validation failure.

## 3. Artifacts generated by default

Each parsed PDF generates the following artifacts by default:

- `meta.json`: document metadata, source, page count, hashes, and PDF path.
- `pages.jsonl`: page-level extracted text, with one page per line.
- `quality_report.json`: text-layer quality, character count by page, and low-text-page assessment.
- `validation_report.json`: structural-completeness, reporting-year, and company-identity validation results for annual reports or prospectuses; generated only for files processed through the high-level source-validation workflow.
- SQLite FTS: the unified full-text search index.

`meta.json`, `pages.jsonl`, and `quality_report.json` are written through temporary files and atomic replacement so an interruption cannot leave partial artifacts. The returned `cache_status` distinguishes `hit`, `miss`, `stale_hash_mismatch`, `stale_missing_hash`, `forced_overwrite`, and `corrupt_cache`.

If the core ingest result is already cached but the current request additionally requires tables, Markdown, full text, or a vector-backend manifest, the Kit generates only the missing artifacts instead of reparsing the entire PDF. The returned `cache_enhanced` and `enhancements_built` fields distinguish a pure cache hit from on-demand enhancement.

## 4. Artifacts not generated by default

The following artifacts are not generated by default:

- `full_text.txt`
- `document.md`
- an embedding vector index (an explicit request generates only a handoff manifest for an external backend)
- full-document OCR output

Rationale:

- `full_text.txt` and `document.md` are primarily for human readers and are unnecessary for machine retrieval.
- Full-document Markdown increases storage use and processing time.
- An LLM normally needs only relevant page-level evidence, not the entire document.

## 5. When Markdown or TXT is generated

These formats are generated only when:

- the user explicitly requests a Markdown export;
- the user explicitly requests a TXT export; or
- after analysis, the user confirms that a human-readable convenience file is needed.

## 6. When OCR is triggered

OCR runs locally and requires neither network access nor an LLM.

With the default `ocr="auto"` setting, the Kit first extracts the PDF's native text. It then applies page-level OCR only to likely scanned pages that contain little text and a high proportion of images, or to pages whose extracted text is genuinely unreadable.

Quality assessment considers character count, readable phrases, the proportion of control or corrupted characters, and the proportion of page area occupied by images. Layout control characters used as cell separators in PDF tables are normalized to spaces first, so a readable page is not classified as corrupted merely because it contains many control characters. Some official PDFs contain large amounts of unusable font-encoded text that appears character-rich but cannot be searched. Such pages still trigger OCR in analysis mode. Download-only mode does not run large-scale OCR for this reason, because doing so would materially slow downloads.

After automatic OCR, the Kit compares the quality of the native text with the OCR text. Native text is replaced only when the OCR result is materially more complete and readable, preventing OCR from reducing the accuracy of tables and numbers. Explicitly setting `ocr="force"` still forces OCR as requested.

If PyMuPDF fails on an individual page, only that page falls back to pypdf. In `quality_report.json`, `extraction_fallback_pages` records pages recovered successfully through fallback, while `extraction_failed_pages` records pages that both parsers failed to process.

Forced OCR is appropriate only for:

- scanned PDFs;
- image-only PDFs;
- PDFs with very poor PyMuPDF extraction quality; or
- explicit user requests for OCR.

## 7. When vectorization is triggered

The current version does not generate embeddings by default.

The default retrieval path is:

```text
SQLite FTS keyword search + general substring fallback search
```

When vector capability is explicitly requested, the current version generates only an `engine=manifest-only` page manifest. It neither generates embeddings nor claims that a searchable vector database has been built. An external vector backend can consume the manifest. If built-in embeddings are added in the future, they should supplement recall rather than replace page-traceable keyword retrieval.

## 8. Retrieval during question answering

Question answering does not send the entire PDF or full-document Markdown to the LLM.

The actual pipeline is:

```text
user question
-> generate keywords and synonyms
-> search with SQLite FTS
-> use substring fallback when necessary
-> read matched pages and neighboring pages
-> assemble an EvidencePacket
-> have the LLM answer from the evidence package
```

## 9. Local cache auditing

By default, `audit_local_pdf_cache_tool(scan_content=false)` checks only file hashes, logical filenames, index references, and stale staging artifacts. With `scan_content=true`, it also validates the content structure of annual reports and prospectuses that can be identified from their filenames. The tool is always read-only and never deletes files automatically. Use a document- or company-level cleanup tool only after confirming an issue.

Content auditing first reconciles the PDF SHA-256 hash, `pages.jsonl`, and the indexed page count. Parsed pages are reused when all three agree, and the PDF is rescanned only when the cache is missing or inconsistent. The returned `content_index_reused_count` and `content_pdf_scanned_count` report the number of documents processed through each path.

File SHA computation uses up to four worker threads, and `hash_workers` reports the number actually used for the current run. Content-structure validation remains deterministic and processes documents one at a time.
